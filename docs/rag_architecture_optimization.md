# HIS RAG 系统架构评估与高级优化方案设计

本项目经过重构后，在开发与运行层面上已完全对齐了**大模型（LLM）与 RAG 领域的业界主流主流技术栈**。以下是针对当前架构的评估，以及后续从“演示版本（Demo）”走向“生产级别（Production）”的深度优化建议。

---

## 📊 1. 当前架构主流度评估

目前系统的核心技术选型是非常典型且主流的 **“轻量化 RAG 解决方案”**：

| 技术层 | 选用技术 | 主流度状态 | 评估与定位 |
| :--- | :--- | :--- | :--- |
| **前端** | Vue 3 + Vite | 🟢 非常主流 | 现代化单页应用（SPA）标配，打包极快，开发体验极佳。 |
| **后端 API** | FastAPI + Uvicorn | 🟢 绝对主流 | Python 社区大模型开发的首选 API 框架，原生支持异步协程与 SSE 流式响应，性能极高。 |
| **LLM 接口** | DeepSeek V3/R1 (OpenAI 协议) | 🟢 行业顶流 | 目前全球性价比与推理能力最强的模型之一。 |
| **向量数据库** | ChromaDB | 🟡 本地主流 | 适合本地、轻量级、嵌入式的 RAG 项目。生产环境（多节点并发）建议升级为独立服务。 |
| **开发框架** | LangChain / LCEL | 🟢 绝对主流 | LLM 应用开发事实上的行业标准，扩展性强。 |
| **结构化数据库** | SQLite | 🟢 合理 | 适合单机环境的会话历史存储，无需安装外部服务。 |

## 🚀 2. 深度架构优化方案（从 Naive RAG 迈向 Advanced RAG）

当前系统处于 **Naive RAG（初级 RAG）** 阶段：即 “切片 -> 向量化 -> 余弦相似度检索 -> 拼接 Prompt -> LLM 生成”。这种设计容易受到干扰数据影响，在医疗场景下，有很大的优化空间。

### 核心优化方向一：混合检索 (Hybrid Search) —— 解决专有名词检索率低的问题
* **痛点**：纯向量检索基于语义，如果医护人员输入了某个罕见的专业药名或拼音缩写，向量模型可能由于训练数据不足而“无法匹配”，但传统的关键词搜索却能百分之百命中。
* **方案**：使用 LangChain 的 **`EnsembleRetriever`** 将 **Chroma 向量检索** 与 **BM25（或 SQLite FTS5）关键词检索** 进行双路召回融合（RRF）。
* **效果**：既保留了模糊语义理解，又保证了专业医学名词的绝对精确匹配。

```text
    [用户问题]
       │
       ├─► [Chroma 向量检索] (语义模糊理解) ──┐
       │                                     ▼
       └─► [BM25 关键词检索] (专有名词匹配) ─► [RRF 倒数排序融合] ─► [融合排名的 Top-K 文档]
```

---

### 核心优化方向二：引入重排模型 (Rerank) —— 提高上下文精准度
* **痛点**：为了防止遗漏，检索器往往会返回 7-10 个切片，但这些切片里混杂了大量无关的干扰噪声。大模型阅读过长的无关上下文，容易产生幻觉（“迷失在中间”现象）。
* **方案**：
  1. 检索阶段（Retrieve）放宽阈值，粗筛召回 15 个切片。
  2. 使用轻量级重排模型（如 `BAAI/bge-reranker-large` 或 Cohere Rerank）对这 15 个切片进行二次精排。
  3. 仅挑选评分最高的前 3-4 个切片输入给大模型。
* **效果**：显著降低大模型 Token 消耗，回答准确率能提升 20% 以上。

---

### 核心优化方向三：高级文档解析与语义切片 (Semantic Chunking)
* **痛点**：现在的切片是按照固定的字符长度（例如 500 字）强行切断的。这经常导致一句话被切成两半，或者表格、临床指南的上下文关系丢失。
* **方案**：
  1. **语义切片 (Semantic Chunking)**：利用 LangChain 的 `SemanticChunker`，它会监控相邻句子之间的 Embedding 差异，只有在语义发生漂移时才进行切片，保证每个 Chunk 都是语义完整的。
  2. **版面还原解析 (Layout-Aware Parsing)**：使用 `Marker` 或 `PyMuPDF` 解析 PDF，提取表格（Markdown Table）并保留排版，防止表格数据被错乱切分。

---

### 核心优化方向四：智能体工作流 (Agentic Workflow) —— 引入 LangGraph
* **痛点**：当前的问答完全依赖单次 RAG 链条。但医疗分诊是一个“诊断-排除-再诊断”的多步决策过程。
* **方案**：引入 **LangGraph**，将问答重构为状态机图。
  * **步骤 1**：LLM 充当“预检分诊” Agent，判断患者描述是否清晰。如果模糊，主动追问（例如：“请问您胸痛是针刺样还是压迫样？”）。
  * **步骤 2**：在判定风险后，如果是“高危红旗”，直接中断图并路由至“紧急救医提示”。
  * **步骤 3**：如果是常规健康问答，调用对应的分诊知识工具库。

```text
    [开始] ──► [症状分析 Agent] ──► {信息是否完整?} ──(否)──► [向用户追问细节]
                                         │                            │
                                        (是)                          ▲
                                         ▼                            │
                                     {风险评估} ──► (常规) ──► [RAG 检索] ──► [输出建议单]
                                         │
                                       (高危)
                                         │
                                         ▼
                                  [急诊红色通道]
```

---

### 核心优化方向五：工程化与可观测性
* **生产数据库升级**：如果将来有多用户并发，单文件 SQLite 和嵌入式 Chroma 将面临读写锁限制。建议生产环境迁移至 **Qdrant / Milvus (向量数据库)** 和 **PostgreSQL (会话历史)**。
* **全链路追踪 (Tracing)**：在项目中集成 **LangSmith** 或 **Phoenix**。这允许您在后台查看每次用户提问的完整调用链：大模型消耗了多少 Token、检索召回了哪些文档、检索得分是多少、大模型的延迟是多少。
